1. CONTEXTO E MOTIVAÇÃO
================================================================================

O sistema de auditoria SPED EFD atingiu a Fase 7 com 121 regras implementadas,
97% de cobertura de testes e arquitetura modular funcional. Uma auditoria técnica
de alto nível realizada em 2026-04-05 identificou lacunas estruturais que impedem
o uso profissional seguro do sistema em produção.

O problema central não é técnico — é de produto e fiscal. O sistema comunica
mais conformidade do que entrega. Um analista que receba "0 erros críticos" em
um arquivo de empresa Simples Nacional com operações interestaduais B2C pode
concluir erroneamente que o arquivo está correto, quando o motor ainda não valida
regime Simples, DIFAL, ST com MVA ou benefícios fiscais estaduais.

Este PRD descreve todos os módulos necessários para fechar essas lacunas,
organizado por prioridade de impacto fiscal e risco.

3. PRINCÍPIOS NÃO NEGOCIÁVEIS
================================================================================

P-01  TODA regra deve ter base normativa citável (artigo, convênio, resolução,
      versão do Guia Prático). Regras sem fundamento citável são no máximo
      warnings do tipo indicio.

P-02  O sistema NUNCA deve modificar um campo fiscal sem confirmação humana
      explícita com justificativa registrada em audit_log.

P-03  O score de conformidade SEMPRE deve exibir, junto ao número, quais
      verificações não foram executadas por falta de contexto ou tabelas.

P-04  Regras dependentes de contexto externo (tabela 5.1.1 por UF, ato
      concessivo, pauta fiscal) devem emitir aviso de "verificação incompleta"
      em vez de falso positivo crítico.

P-05  Metarregras de governança do sistema não são erros do arquivo SPED.
      Devem ser apresentadas em seção separada do relatório.

P-06  O relatório exportado deve ter rodapé de responsabilidade obrigatório,
      listando limitações conhecidas do motor e exigindo validação profissional.

P-07  Toda correção aplicada deve ser rastreável: quem aprovou, quando, qual
      justificativa, qual regra gerou o apontamento, valor original e novo valor.

Empresas do Simples Nacional usam CSTs 101-900 (Tabela B). Aplicar Tabela A

    51  Diferimento
    60  ICMS cobrado anteriormente por ST
    70  Redução de base + ST
    90  Outras

MÓDULO 2 — CORREÇÃO DE LÓGICA NO C190
================================================================================

ID:         MOD-02
Prioridade: CRÍTICA (erro de lógica direta que causa falsos negativos)
Fase:       1
Módulo:     src/validators/intra_register_validator.py
            src/validators/c190_validator.py

2.1 PROBLEMA
As regras C190_SOMA_VL_OPR, C190_SOMA_VL_BC_ICMS e C190_SOMA_VL_ICMS agrupam
C170 apenas por CFOP ao comparar com C190. O Guia Prático EFD v3.2.2 define
que o C190 agrupa por CST_ICMS + CFOP + ALIQ_ICMS (três campos).

Um documento com dois itens de CFOP 5102, um com CST 00 (alíq 18%) e outro
com CST 40 (isento), terá dois C190 diferentes. O sistema atual soma ambos
e compara com o primeiro C190 encontrado — resultado: falso negativo garantido.

2.2 CORREÇÃO NECESSÁRIA

  ANTES (errado):
    chave_agrupamento = cfop
    soma_c170 = sum(c170.VL_ITEM for c170 in itens if c170.CFOP == c190.CFOP)

  DEPOIS (correto):
    chave_agrupamento = (cst_icms, cfop, aliq_icms)
    soma_c170 = sum(
        c170.VL_ITEM for c170 in itens
        if (c170.CST_ICMS, c170.CFOP, c170.ALIQ_ICMS)
        == (c190.CST_ICMS, c190.CFOP, c190.ALIQ_ICMS)
    )

2.3 REGRAS AFETADAS
  C190_SOMA_VL_OPR        → corrigir agrupamento
  C190_SOMA_VL_BC_ICMS    → corrigir agrupamento
  C190_SOMA_VL_ICMS       → corrigir agrupamento
  C190_001 (c190_validator) → corrigir agrupamento
  cross_block_validator C vs E → verificar se usa o mesmo agrupamento

2.4 CRITÉRIOS DE ACEITAÇÃO
  [ ] Agrupamento triplo (CST+CFOP+ALIQ) implementado em todos os validators
  [ ] Teste: documento com 2 CSTs no mesmo CFOP → dois C190 validados corretamente
  [ ] Teste: documento com CST 40 + ALIQ 0 → C190 correspondente aceito
  [ ] Regressão: todos os 47 testes intra-registro continuam passando

MÓDULO 3 — SISTEMA DE TOLERÂNCIAS PARAMETRIZADAS
================================================================================

ID:         MOD-03
Prioridade: ALTA
Fase:       1
Módulo:     src/validators/tolerance.py + rules.yaml

3.1 PROBLEMA
A tolerância global `tolerance: 0.02` é aplicada a todos os recálculos. Para
VL_IPI de R$ 0,05, permite 40% de erro. Para E110 com VL_TOT_DEBITOS de
R$ 800 mil, é razoável. Para C190 com 200 itens, o arredondamento acumulado
pode facilmente superar R$ 0,02 sem qualquer erro real.

3.2 ESTRUTURA DE TOLERÂNCIAS

  src/validators/tolerance.py

  TOLERANCES = {
      # Comparações item a item (C170)
      "item_icms":        0.01,   # BC * ALIQ/100 vs VL_ICMS por item
      "item_ipi":         0.01,   # BC * ALIQ/100 vs VL_IPI por item
      "item_pis":         0.01,   # BC * ALIQ/100 vs VL_PIS por item
      "item_cofins":      0.01,   # BC * ALIQ/100 vs VL_COFINS por item
      # Comparações de documento (C100)
      "doc_vl_doc":       0.02,   # Soma de componentes vs VL_DOC
      # Comparações de consolidação (C190 vs soma C170)
      "consolidacao":     lambda n: max(0.02, 0.01 * (n ** 0.5)),
                          # n = número de itens. Para 100 itens: 0,10
      # Comparações de apuração (E110)
      "apuracao_e110":    0.05,   # Soma C190+D690 vs E110
      # Comparações de inventário (H010)
      "inventario":       0.10,
  }

  def get_tolerance(comparison_type: str, n_items: int = 1) -> float:
      tol = TOLERANCES[comparison_type]
      return tol(n_items) if callable(tol) else tol

ID:         MOD-05
Prioridade: CRÍTICA
Fase:       1

5.1 PROBLEMA
Cada validator acessa o banco de forma independente para buscar informações
do arquivo (UF, período, regime, etc.). Não há objeto centralizado de contexto
passado entre os stages do pipeline. Isso causa:
- Consultas redundantes ao banco
- Impossibilidade de regras condicionais (ex: "só aplica se regime = normal")
- Dificuldade de testar validators sem banco real

5.2 ESTRUTURA DO VALIDATIONCONTEXT

  @dataclass
  class ValidationContext:
      file_id: int
      filename: str
      hash_sha256: str

      # Extraído do 0000
      regime: TaxRegime              # NORMAL | SIMPLES_NACIONAL | MEI | UNKNOWN
      ind_perfil: str                # A | B | C
      cod_ver: str
      uf_contribuinte: str
      cnpj: str
      periodo_ini: date
      periodo_fim: date
      company_name: str

      # Tabelas de referência disponíveis
      available_tables: set[str]     # {"matriz_aliq_uf", "ncm_tipi", ...}

      # Cache de cadastros (populado no parse)

      produtos: dict[str, dict]       # COD_ITEM → {NCM, ...}

Regras:     DIFAL_001 a DIFAL_008

6.1 CONTEXTO LEGAL
EC 87/2015: obriga recolhimento de DIFAL nas operações interestaduais
            destinadas a consumidor final não contribuinte de ICMS.
            Vigência: 01/01/2016 (com escalonamento de partilha até 2019).
EC 190/2022: define que o DIFAL pertence ao estado de destino.
             ADI 5469: declarou inconstitucional a cobrança sem lei complementar.
             LC 190/2022: regularizou a cobrança a partir de 01/01/2022.
FCP: Fundo de Combate à Pobreza — alíquota adicional ao DIFAL, varia por UF.

6.2 IDENTIFICAÇÃO DA OBRIGATORIEDADE
DIFAL é obrigatório quando:
  a) CFOP começa com 6 (interestadual)
  b) UF_DEST != UF do emitente (0000.UF)
  c) Destinatário é consumidor final não contribuinte
     (IE vazia, ou IE = "ISENTO", e PF/PJ sem inscrição estadual ativa)
  d) Mercadoria não tem isenção específica ou ST que cubra o diferencial

Indicadores no SPED:
  - C170.CFOP iniciado por 6
  - 0150.IE vazia ou ISENTO para o COD_PART
  - C170.CST_ICMS = 00, 20, 40 (ou equivalente Simples Nacional)
  - E300 (Bloco E — Apuração DIFAL) ausente ou inconsistente

6.3 REGRAS A IMPLEMENTAR

     Definido por portaria estadual, por produto

  c) Preço final ao consumidor sugerido pelo fabricante:

9.1 PROBLEMA
fields_json armazena campos como array JSON posicional: ["val1","val2","val3"].
O acesso é feito por índice (fields[2] para o terceiro campo). Se o parser
contou incorretamente os pipes, ou se a definição de tamanho de campo mudou,
o índice fica silenciosamente errado.

9.2 SOLUÇÃO: DICIONÁRIO NOMEADO

  ANTES (frágil):
    fields_json = '["SP","01012024","31012024","EMPRESA X",...]'
    uf = json.loads(record.fields_json)[2]   # posição magic number

  DEPOIS (robusto):
    fields_json = '{"UF":"SP","DT_INI":"01012024","DT_FIN":"31012024","NOME":"EMPRESA X"}'
    uf = json.loads(record.fields_json)["UF"]  # nome explícito

9.3 IMPLEMENTAÇÃO NO PARSER

  parser.py:
    def parse_line(line: str, field_defs: list[FieldDef]) -> dict:
        values = line.strip().split("|")
        # values[0] = "" (antes do primeiro pipe)
        # values[1] = "registro" (ex: C100)
        # values[2..n] = campos
        result = {}
        for i, field in enumerate(field_defs):
            idx = i + 2  # pula separador inicial e nome do registro
            result[field.name] = values[idx] if idx < len(values) else ""
        return result

10.1 PROBLEMA
O auto_correction_service.py pode aplicar correções e o export_service.py
exporta o SPED corrigido. O SPED EFD é um documento com validade jurídica.
Qualquer modificação sem aprovação de profissional habilitado é risco legal
para o usuário e para o desenvolvedor do sistema.

10.2 REGRAS INVIOLÁVEIS (PRINCÍPIO P-02)
  a) NUNCA aplicar correção sem confirmação explícita do usuário
  b) NUNCA oferecer "Corrigir todos" para erros não-determinísticos
  c) TODA correção deve ter justificativa textual registrada
  d) O export de SPED corrigido deve exigir confirmação final
  e) Correções de CST, CFOP, alíquota NUNCA devem ser sugeridas como valores
     prontos para aplicação — apenas explicadas textualmente

10.3 CLASSIFICAÇÃO DE CORREÇÕES PERMITIDAS E PROIBIDAS

  PERMITIDAS (determinísticas, sem ambiguidade jurídica):
    - Formato de data inválido → corrigir para DDMMAAAA se valor claro
    - CNPJ com zeros extras → remover caracteres não numéricos
    - Campo numérico com vírgula decimal em vez de ponto
    - Chave NFe com tamanho errado se o conteúdo é unívoco

  PERMITIDAS COM AVISO (matemáticas, mas com implicação fiscal):
    - VL_ICMS recalculado difere em 0,01 → sugerir valor calculado
      (exige confirmação + justificativa)

  PROIBIDAS (nunca sugerir valor pronto, apenas orientação):
    - Troca de CST
    - Troca de CFOP
    - Alteração de alíquota
    - Qualquer campo do E110, E111, E112, E113
    - Qualquer campo do bloco H (inventário)
    - Qualquer campo de benefício fiscal

10.4 FLUXO DE APROVAÇÃO

  API: PUT /api/files/{id}/records/{rec_id}
    Request body:
      field_no: int
      field_name: str

MÓDULO 13 — ESCALA ORTOGONAL DE ACHADOS
================================================================================

ID:         MOD-13
Prioridade: MÉDIA
Fase:       2
Módulo:     src/models.py + validation_errors schema + frontend

13.1 PROBLEMA
A escala única critical/error/warning/info cobre dimensões ortogonais:
certeza do achado (é realmente um erro?) e impacto fiscal (quanto importa?).
Um indício de benefício fiscal (certeza baixa, impacto potencialmente alto)
aparece igual a um erro de formato de data (certeza alta, impacto baixo).

13.2 MODELO PROPOSTO

  Dimensão 1 — CERTEZA:
    objetivo    → comprovado matematicamente ou por campo inválido
    provavel    → lógica fiscal aponta erro mas depende de contexto
    indicio     → padrão suspeito, requer investigação

  Dimensão 2 — IMPACTO:
    critico     → risco de autuação, recolhimento a menor, passivo fiscal
    relevante   → inconsistência que pode gerar questionamento
    informativo → observação técnica sem impacto imediato

  Mapeamento de compatibilidade com severidade atual:
    objetivo + critico    → mantém critical
    objetivo + relevante  → mantém error
    provavel + critico    → mantém error (com flag de contexto faltando)
    provavel + relevante  → warning
    indicio + qualquer    → warning (nunca error ou critical sem certeza)
    metarregra            → info

13.3 IMPLEMENTAÇÃO
  a) Adicionar campos à tabela validation_errors:
       certeza TEXT   -- "objetivo" | "provavel" | "indicio"
       impacto TEXT   -- "critico" | "relevante" | "informativo"

  b) Adicionar campos ao schema de regra no rules.yaml:
       certeza: objetivo
       impacto: critico

  c) Frontend: filtros separados por certeza e impacto
     Analista pode ver: "todos os objetivos + críticos" (ação imediata)
                        "todos os indícios + críticos" (investigação)
                        "tudo warning" (revisão posterior)

  [ ] Campos certeza e impacto em validation_errors

16.1 PROBLEMA

modela a relação entre uma escrituração original (0000.COD_VER = 0) e
suas retificações (COD_VER = 1, 2, 3, ...).

Validar uma retificação sem a original pode gerar:
- Falso positivo: ajuste no E111 que só faz sentido como correção da original
- Falso negativo: inconsistência que só aparece comparando as duas versões

16.2 MODELO DE DADOS

  Alterações na tabela sped_files:
    cod_ver INTEGER DEFAULT 0          -- versão do arquivo (0=original)

  Nova tabela: sped_file_versions
    id INTEGER PRIMARY KEY
    original_file_id INTEGER NOT NULL
    retificador_file_id INTEGER NOT NULL
    cod_ver INTEGER NOT NULL
    FOREIGN KEY (original_file_id) REFERENCES sped_files(id)
    FOREIGN KEY (retificador_file_id) REFERENCES sped_files(id)

16.3 VALIDAÇÕES DE RETIFICAÇÃO

  RET_001 — Retificador com período diferente da original
    Condição: DT_INI ou DT_FIN diferente entre original e retificador
    Severidade: error

    Condição: COD_VER > 0 e nenhum arquivo com COD_VER = 0 para o mesmo
              CNPJ + período
    Severidade: warning (informativo — a original pode estar em outro sistema)

  RISCO-02  Usuário exporta SPED corrigido com erro de interpretação fiscal
    Impacto: máximo (passivo fiscal + autuação)
    Probabilidade: alta sem MOD-10
    Mitigação: MOD-10 + aviso legal obrigatório

  RISCO-03  Score 0 erros = crença de conformidade total
    Impacto: alto (decisão errada baseada em dados incompletos)
    Probabilidade: certa sem MOD-11
    Mitigação: MOD-11 exibido sempre que cobertura < 100%

  RISCO-04  Tabelas de referência desatualizadas
    Impacto: alto (alíquotas incorretas, municípios extintos)
    Probabilidade: média a longo prazo
    Mitigação: versionar tabelas com data de vigência (MOD-04 e MOD-14)
               e exibir data de última atualização no AuditScopePanel

  RISCO-05  Performance em arquivos > 50 mil registros
    Impacto: médio (timeouts, UX degradada)
    Probabilidade: média (arquivos grandes são comuns em indústrias)
    Mitigação: streaming de parsing (MOD-19) + paginação + SSE já implementado

  RISCO-06  Modelo de embeddings muda e degrada busca semântica
    Impacto: médio (sugestões menos relevantes)
    Probabilidade: baixa no curto prazo
    Mitigação: versionamento do modelo no banco (MOD-19)

  RISCO-07  Acúmulo de regras sem base legal documentada
    Impacto: alto (regras erradas passam despercebidas)
    Probabilidade: média (risco de crescimento acelerado do catálogo)
    Mitigação: Princípio P-01 inviolável: toda nova regra exige campos
              `legislation` e `vigencia_de` preenchidos no rules.yaml

28. DEFINIÇÕES E GLOSSÁRIO
================================================================================

SPED EFD   Sistema Público de Escrituração Digital — Escrituração Fiscal
           Digital. Arquivo .txt pipe-delimited enviado mensalmente à SEFAZ
           contendo todos os documentos fiscais, apurações e inventários.

Guia Prático
           Manual oficial da SPED EFD ICMS/IPI publicado pelo ADE COTEPE/ICMS.
           Versão de referência: 3.2.2 (verificar versão vigente).

CST        Código de Situação Tributária. Define como o ICMS se aplica ao item.
           Tabela A: Regime Normal (CSTs 00-90).
           Tabela B: Simples Nacional (CSTs 101-900).

CFOP       Código Fiscal de Operações e Prestações. 4 dígitos. Primeiro dígito:
           1=entrada interna, 2=entrada interestadual, 3=entrada exterior,
           5=saída interna, 6=saída interestadual, 7=saída exterior.

DIFAL      Diferencial de Alíquota. ICMS devido na diferença entre a alíquota
           interna do estado destino e a alíquota interestadual. Obrigatório
           desde EC 87/2015 para operações B2C interestaduais.

E110       Registro de apuração do ICMS. Totaliza débitos, créditos e ajustes
           do período. O saldo determina o valor a recolher ou a crédito.

E111/E112/E113  Registros de ajuste de apuração. E111=ajuste geral,
               E112=informações adicionais do ajuste, E113=documentos que
               fundamentam o ajuste. Usados para registrar benefícios fiscais.

MVA        Margem de Valor Agregado (também chamado IVA-ST). Percentual
           aplicado sobre o valor da operação para calcular a base de
           cálculo do ICMS-ST. Definido por protocolo/convênio por NCM e UF.

Pauta Fiscal
           Tabela de preços máximos ao consumidor definida pelo Fisco estadual,
           usada como base de cálculo do ICMS-ST em alguns produtos.

FCP        Fundo de Combate à Pobreza. Percentual adicional ao ICMS cobrado
           por alguns estados em determinadas operações. Varia por UF.

COD_VER    Campo do registro 0000. Indica a versão do arquivo:
           0 = escrituração original, 1+ = retificadores.

IND_PERFIL Campo do registro 0000. Indica o perfil da escrituração:
           A = perfil completo (Regime Normal), B = simplificado,
           C = Simples Nacional ou ME/EPP.

Tipo_achado
           Classificação da natureza do achado pela regra:
           erro_objetivo: provado matematicamente ou por campo inválido
           inconsistencia: incoerência lógica entre campos
           indicio: padrão suspeito que requer investigação
           risco_fiscal: exposição a passivo sem erro formal confirmado
           metarregra: regra sobre o processo de auditoria, não o arquivo

Certeza    Dimensão ortogonal à severidade:
           objetivo: sem ambiguidade, comprovado pelos dados do arquivo
           provavel: lógica fiscal forte, mas depende de contexto externo
           indicio: padrão suspeito, conclusão não determinística

Impacto    Segunda dimensão ortogonal:
           critico: risco de autuação, passivo fiscal relevante
           relevante: questionamento técnico, correção recomendada
           informativo: observação sem consequência imediata
