PARSING DO XML
──────────────

  Lib: xml.etree.ElementTree (stdlib)

  Cuidados obrigatórios:
    - Tratar namespace XML da NF-e (xmlns="http://www.portalfiscal.inf.br/nfe")
    - Aceitar <nfeProc> (com protocolo) e <NFe> (sem protocolo)
    - Campos ausentes → None/0.0 (não crashear)
    - XML malformado → status='invalid_xml', logar, continuar batch
    - CST do item: se CRT emitente = 1 (SN) → usar CSOSN; se CRT = 3 → usar CST

  Fluxo:
    1. Detectar envelope (<nfeProc> ou <NFe>)
    2. Ler <infNFe> → identificação, emitente, destinatário
    3. Ler <det> → itens com produto + tributos
    4. Ler <total><ICMSTot> → totalizadores
    5. Ler <protNFe> → protocolo (cStat, chave)
    6. Normalizar todos os campos (Etapa 0)
    7. Salvar em nfe_xmls (cabeçalho) + nfe_itens (detalhe)

18. PLANO — IA para Tratamento de Erros (cache incremental)
================================================================================

STATUS: PLANEJADO (não implementado)
PRINCÍPIO: IA nunca cria regras fiscais. Apenas trata a apresentação dos erros.

ARQUITETURA DE CACHE
────────────────────

  Chave ampliada (evita colisão de contexto):
    (rule_id, regime, uf, beneficio_codigo, ind_oper, campo_principal)

  Motivo: dois erros do mesmo error_type podem exigir explicações diferentes
  conforme bloco/registro, severidade, entrada vs saída, campo envolvido.

  Valor: texto gerado pela IA

  Fluxo:
    1. Erro gerado pelo validator → montar chave do cache
    2. Buscar no banco (ai_error_cache) por chave_hash
    3. Se encontrar → usar texto cacheado (zero IA, incrementar hits)
    4. Se não encontrar → chamar IA → salvar no cache → usar
  Após 2-3 meses: 99% dos erros já cacheados, IA quase nunca chamada


FRENTES
───────

  Frente 1 — Explicação Inteligente (maior impacto)
    Input: erro bruto + contexto (regime, UF, benefícios, CNAE)
    Output: explicação personalizada em português para contador
    Onde: complementar _enrich_errors() no pipeline
    Modelo: Claude Haiku (rápido, barato) via batch

  Frente 2 — Agrupamento e Priorização
    Input: lista de 2000+ erros de um SPED
    Output: 3-5 problemas principais com ordem de resolução
    Onde: novo endpoint POST /api/files/{id}/ai-summary
    Modelo: Claude Haiku

  Frente 3 — Sugestão de Correção Contextual
    Input: erro sem expected_value + contexto
    Output: sugestão textual de como corrigir
    Onde: campo ai_suggestion na tabela validation_errors
    Modelo: Claude Haiku

  Frente 4 — Relatório Narrativo
    Input: todos os erros + contexto do contribuinte
    Output: parecer de fechamento em linguagem contábil/fiscal
    Onde: novo endpoint GET /api/files/{id}/report/narrative
    Modelo: Claude Sonnet (texto longo, qualidade)

19. PLANO — Auditoria Unificada (Pipeline Unico SPED + XML + Regime + Beneficio)
================================================================================

STATUS: PLANEJADO (não implementado)
IMPACTO: Arquitetural — elimina revalidação, unifica fluxo, replica lógica da RFB
REFERÊNCIA: Procedimentos de auditoria fiscal da Receita Federal (malha ICMS)

PROBLEMA ATUAL
──────────────

  Hoje existem 2 processos independentes que competem entre si:

  1. Pipeline SPED (26 validadores)
     - Roda estrutural → cruzamento blocos → enriquecimento
     - DELETE todos os validation_errors ao iniciar
     - Não sabe que existem XMLs

  2. Cruzamento XML (17 regras XML001-XML017)
     - Roda separadamente, em momento diferente
     - Gera erros com categoria='cruzamento_xml'
     - O pipeline SPED apaga esses erros ao revalidar

  Consequências:
    - O usuário precisa executar 2 processos manualmente
    - Se revalida o SPED, perde os erros do XML
    - Os validadores de recálculo não usam dados do XML como referência
    - A Receita Federal não faz assim

COMO A RECEITA FEDERAL FAZ
──────────────────────────

  A RFB tem 4 fontes de dados e cruza TUDO JUNTO em uma passada:

  ┌─────────────────┐    ┌─────────────────┐
  │  XML NF-e       │    │  SPED EFD       │
  │  (SEFAZ)        │    │  (Contribuinte) │
  │  FONTE VERDADE  │    │  DECLARAÇÃO     │
  └────────┬────────┘    └────────┬────────┘
           │                      │
           ▼                      ▼
  ┌────────────────────────────────────────┐
  │       AUDITORIA UNIFICADA             │
  │                                        │
  │  1. Validação estrutural do SPED       │
  │  2. Cruzamento entre blocos SPED       │
  │  3. Cruzamento XML x SPED             │◄── NOVO ESTÁGIO
  │  4. Recálculo com dados XML+regime     │
  │  5. Enriquecimento e base legal        │
  │                                        │
  └────────────────────────────────────────┘
           │                      │
           ▼                      ▼
  ┌─────────────────┐    ┌─────────────────┐
  │  Regime Tribut. │    │  Benefícios     │
  │  (Cadastro)     │    │  (Concessões)   │
  └─────────────────┘    └─────────────────┘

  Hierarquia de confiança:
    XML NF-e > Cadastro RFB > SPED EFD > Declarações acessórias

  Quando há divergência entre XML e SPED:
    - O XML é a fonte da verdade (veio da SEFAZ, tem protocolo)
    - O SPED é a declaração do contribuinte (pode ter erro)
    - O regime determina quais regras se aplicam
    - O benefício determina se isenção/redução é legítima

SOLUÇÃO: PIPELINE ÚNICO COM ESTÁGIO XML INTEGRADO
──────────────────────────────────────────────────

  O cruzamento XML vira o estágio 2.5 do pipeline existente.
  Os 26 validadores continuam intactos.
  Se não tem XMLs, o estágio é pulado.

  Novo fluxo de estágios:

  ┌──────────────┐
  │ ESTRUTURAL   │ Formatos, campos obrigatórios, tipos
  │ (9+4+10=23)  │
  └──────┬───────┘
         ▼
  ┌──────────────┐
  │ CRUZAMENTO   │ C100 x C170 x C190, refs 0150/0200,
  │ SPED (100+)  │ recálculo, CST, semântica, alíquotas
  └──────┬───────┘
         ▼
  ┌──────────────┐
  │ CRUZAMENTO   │ XML001-XML017: presença, valores,    ◄── NOVO
  │ XML (17)     │ itens, CNPJ, datas, canceladas
  └──────┬───────┘
         ▼
  ┌──────────────┐
  │ ENRIQUECIM.  │ Mensagens amigáveis, base legal,
  │              │ auto-corrigibilidade
  └──────────────┘
